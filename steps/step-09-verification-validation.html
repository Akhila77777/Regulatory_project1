Verification, validation & traceability

This is where RA meets your engineering background directly: turning intended use and risks into testable requirements, then proving each one. Files: docs/software/srs.md, architecture.md, traceability_matrix.md, verification_validation/.

The golden thread

Everything must trace, both ways:

User need / Intended use → Requirement (REQ) → Design / Architecture → Test (TST)
Risk (RISK) → Risk control → Requirement (REQ) → Verification (TST)

No requirement without a source; no risk control without a verifying test; no test without a requirement. That is what the traceability matrix enforces.

Writing a good requirement (SRS)

Reg / good practice - requirement quality
A requirement must be unambiguous, testable, atomic, and traceable. Use "shall". Bad: "the model should be accurate." Good: "REQ-014: The system shall achieve sensitivity >= 0.90 (95% CI lower bound >= 0.85) for melanoma on the held-out test set, reported per Fitzpatrick group." Each REQ records: type, text, rationale, source, and linked test.

Requirement types to cover

  • Functional - accepts an image, returns a malignancy probability + disclaimer.
  • Performance - sensitivity/specificity/latency targets.
  • Safety - rejects out-of-distribution inputs; warns on low confidence (these come from RISK items).
  • Usability - output is unambiguous, oversight-supporting (IEC 62366-1).
  • Security - authentication, input validation, logging (links to cybersecurity).
  • Regulatory - logging per AI Act Art. 12; transparency per Art. 13.

Tip - this is your interview superpower
Your test-engineering background maps 1:1 onto V&V. In interviews, explicitly connect them: "I am comfortable owning the traceability from requirements to tests because that is what I did as a test engineer; the regulatory layer adds the risk and intended-use sources, which I have practised on DermaScan."

Special V&V challenges for ML

  • Non-determinism - fix seeds, pin library versions (SOUP), record the exact model artifact hash.
  • Reproducibility - the test must be re-runnable: dataset version + code version + environment.
  • Metamorphic & robustness testing - check invariance to small, label-preserving image perturbations (rotation, lighting, compression).
  • Regression on retrain - a fixed benchmark suite that every new model version must pass before release.
Hot 2026 - "show your test rig"
Employers increasingly want to see automated, version-controlled V&V pipelines (CI that runs the metric suite, OOD checks, and subgroup fairness gates on every model change). Building even a small one in DermaScan is a concrete, demonstrable skill.

Interview Q&A

What makes a good requirement?

Unambiguous, testable, atomic, and traceable, written with "shall", with a measurable acceptance criterion, a rationale, a source (intended use, risk control, or regulation), and a linked verification.

What is a traceability matrix and why does it matter?

A table linking user needs/risks to requirements to design to tests, both forward and backward. It proves coverage: every requirement is verified, every risk control is tested, and nothing is orphaned. It is core evidence in a Notified Body audit.

How do you verify and validate an ML model differently from normal software?

You add reproducibility controls (seeds, pinned versions, artifact hashing), evaluate on an independent test set with subgroup and robustness/metamorphic testing, guard against data leakage, and run a regression benchmark suite on every retrain, alongside clinical validation with users for the validation side.

How does a risk control become testable?

The control is written as a requirement (e.g. "shall reject out-of-distribution inputs"), implemented in design, and verified by a specific test, with the chain recorded in the traceability matrix so the control's effectiveness is demonstrably evidenced.
