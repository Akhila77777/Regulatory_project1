IEC 62304 software lifecycle

IEC 62304 defines the lifecycle processes for medical device software. The amount of process rigour scales with the software safety class. Project folder: docs/regulatory/iec62304/.

Software safety classification (A / B / C)

The class depends on the severity of harm that the software could contribute to after considering risk controls outside the software:

ClassPossible harm from a software failureRigour
ANo injury or damage to health is possibleLightest
BNon-serious injury is possibleMedium
CDeath or serious injury is possibleHeaviest
Reg - the class is set by risk, not assumed
You assign the class from the hazard analysis (ISO 14971). Crucially, you may assign a lower class if an external risk control (a hardware safeguard, or a procedural one like mandatory clinician confirmation) reduces the possible harm. The decision and its justification must be documented.
Tip - DermaScan's class is a judgement call (talk about it)
A missed melanoma is a serious harm, which argues for Class C. But if you credibly position DermaScan as decision-support with mandatory clinician review, you can argue the software alone cannot cause serious harm, supporting Class B. The "right" interview answer is: I would derive it from the risk analysis and document the rationale; I would not default to B just to reduce work. Do not state a class in the project without the written argument.

The lifecycle processes

  1. Software development planning (5.1) - the SDP: lifecycle model, activities, deliverables, tooling.
  2. Requirements analysis (5.2) - software requirements (functional, performance, safety, interfaces).
  3. Architectural design (5.3) - decompose into items; identify SOUP.
  4. Detailed design (5.4) - for Class C.
  5. Implementation & unit verification (5.5).
  6. Integration & integration testing (5.6).
  7. System testing (5.7).
  8. Release (5.8) - known anomalies documented and assessed.

Plus the continuous "supporting" processes: risk management (clause 7, ties to ISO 14971), configuration management (clause 8), problem resolution (clause 9), and maintenance (clause 6).

SOUP - Software of Unknown Provenance

SOUP is any software item not developed under IEC 62304 for this device - i.e. third-party and open-source libraries. For DermaScan that is PyTorch/TensorFlow, FastAPI, NumPy, Pillow, etc.

Reg - SOUP requirements
For each SOUP item you must: identify it (name + version), state its functional/performance requirements and the hardware/software it needs, and evaluate its known anomalies (e.g. published bug lists / CVEs) for relevance to safety. Maintain a SOUP list. In this project use IDs SOUP-XXX and pin versions in requirements.txt.

Where ML breaks the classic model

IEC 62304 predates modern ML and does not directly address training data, model versioning, or non-determinism. Bridge it with:

  • AAMI CR34971 / BS/AAMI 34971 - guidance on applying ISO 14971 to AI/ML.
  • ISO/IEC 42001 - AI management system (the "QMS for AI").
  • Treat the trained model + its training/validation datasets as configuration items under version control; document data and model lineage.
Hot 2026 - lifecycle for continuously-learning AI
Regulators are focused on change management for AI: what happens when you retrain? The expectation is a predetermined change control plan (the EU is converging with the FDA concept of a "Predetermined Change Control Plan / PCCP") so that defined model updates do not each require a new conformity assessment. Mention this; it is current and senior-sounding.

Interview Q&A

How is IEC 62304 software safety class determined?

From the hazard analysis: Class A if no injury is possible, B if non-serious injury is possible, C if death or serious injury is possible. You may justify a lower class when an external risk control (hardware or procedural) limits the harm the software alone can cause. The decision must be documented.

What would you classify DermaScan as, B or C?

It depends on the risk analysis. A missed melanoma is serious harm, pointing to C; but if mandatory independent clinician review is a credible external control, the software alone may not cause serious harm, supporting B. I would document the rationale rather than assume a class to save effort.

What is SOUP and how do you handle it?

Software of Unknown Provenance - third-party/open-source components not developed to IEC 62304. You identify each (name + version), specify its requirements, and assess its known anomalies/CVEs for safety relevance, keeping a maintained SOUP list and pinned versions.

How do you manage software changes after release?

Through the maintenance, configuration management, and problem-resolution processes: assess each change for impact on safety and risk, re-verify affected items, and update documentation and the risk file. For AI, a predetermined change control plan can pre-authorise defined retraining within limits.
